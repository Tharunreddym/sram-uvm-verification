# Real Simulation Runbook

Use this file when you are ready to produce portfolio-grade evidence. The rule is simple:

> `reports/regression_results.json` must come only from actual simulator logs.

The regression script enforces this by deleting stale dashboard inputs at the start of a run and refusing to write `regression_results.json` if required tools are missing or compile/elaboration fails before seed simulations.

## 1. Check simulator availability

From the project root:

```bash
which vlog vsim vcover      # Questa
which vcs urg              # VCS
which xrun                 # Xcelium
```

At least one supported simulator path must resolve.

## 2. Run a quick real smoke regression

```bash
make clean
make regression SIM=questa TEST=base_test SEEDS=1
make verify
```

Expected real output after success or a real design/testbench failure:

```text
reports/logs/questa_compile_step1.log
reports/logs/questa_compile_step2.log
reports/logs/questa_base_test_seed_1.log
reports/regression_results.json
reports/regression_snapshot.html
```

A failing real simulation is still credible evidence if the log is real. A manually authored pass/fail table is not.

## 3. Run the full 20-seed random regression

```bash
make clean
make regression SIM=questa TEST=random_test SEEDS=20
make verify
```

For VCS:

```bash
make clean
make regression SIM=vcs TEST=random_test SEEDS=20
make verify
```

For Xcelium:

```bash
make clean
make regression SIM=xcelium TEST=random_test SEEDS=20
make verify
```

## 4. What to commit after a real run

For a portfolio repo, commit these only after they were generated by a real run:

```text
reports/regression_results.json
reports/regression_snapshot.html
reports/logs/*.log
reports/ucdb/*.ucdb                    # Questa, if generated
reports/questa_coverage_html/          # Questa, if generated
reports/vcs_cov.vdb/                   # VCS, if generated
reports/vcs_urg_report/                # VCS, if generated
reports/xcelium_cov/                   # Xcelium, if generated
```

## 5. What not to do

Do not manually edit these fields:

```json
"status"
"coverage_pct"
"duration_s"
"uvm_errors"
"uvm_fatals"
"coverage_db"
```

Do not add fake notes like “coverage derived from static analysis.” That instantly weakens the project.

## 6. If tools are missing

The script writes:

```text
reports/preflight_status.json
```

It does **not** write:

```text
reports/regression_results.json
```

That is intentional. It prevents the dashboard from displaying stale or fake results.
